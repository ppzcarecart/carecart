# Carecart partner-app sample (Flutter)

A reference Flutter app that hosts carecart inside a WebView the way a
production partner app would. It exists to verify two things end-to-end
on a real device before the production team commits to one of the
integration paths:

1. The **Home** button on carecart's mobile bottom nav successfully
   closes the WebView and lands the user back in the host app — across
   all three carecart-supported bridges.
2. **Manage Collection's** QR scanner can actually open the camera
   from inside the WebView (Android requires the host app to forward
   the permission grant — getting that wrong is silent).

## What's in this folder

```

├── pubspec.yaml             // Flutter manifest + deps + launcher-icon config
├── analysis_options.yaml    // lint config
├── assets/
│   ├── icon.png             // 1024×1024 carecart icon master
│   └── icon-fg.png          // adaptive-icon foreground (transparent)

│   ├── home_screen.dart     // Shop tile + settings link
│   ├── settings_screen.dart // form for ppzId / email / baseUrl / closeUrl
│   ├── settings_store.dart  // shared_preferences wrapper
│   └── webview_screen.dart  // embeds carecart, intercepts close URL,
│                            // grants camera permission, shows splash
├── android/                 // generated by flutter create — committed
├── ios/                     // generated by flutter create — committed
└── README.md
```

The `android/` and `ios/` folders are committed so a clone-and-run is
one command. They include the manifest entries the integration needs
(`CAMERA` + `uses-feature` on Android, `NSCameraUsageDescription` on
iOS) and the per-density launcher icons generated from `assets/`.

## Prerequisites

1. **Flutter SDK** on your PATH:
   <https://docs.flutter.dev/get-started/install>
2. **Android Studio** with the Flutter and Dart plugins.
3. An **Android emulator** (Tools → Device Manager → Create device) or
   a physical Android device with USB debugging enabled. To test the
   camera scanner, use a physical device — emulator camera support is
   inconsistent.

Verify the toolchain:

```bash
flutter doctor
```

Anything in red should be fixed before continuing.

## Setup

```bash
cd partner-app-sample
flutter pub get
```

That's it — `android/` and `ios/` are already in the tree.

If you ever need to regenerate the platform projects from scratch
(e.g. you bumped Flutter and want fresh defaults), delete the folders
and re-run:

```bash
flutter create --org com.example --project-name carecart_partner_sample .
flutter pub get
```

After regenerating you'll need to re-add the camera manifest entries
(see [Camera permission](#camera-permission) below).

### Regenerate launcher icons (optional)

were generated from `assets/icon.png` (full mark) and

## Using the app

1. Home screen shows a green **Shop** tile and a cog icon (top right).
2. Tap the cog → fill in **PPZ ID**, **Email**, leave **Base URL** as
   `https://web-production-ae5ea.up.railway.app`, leave **Close URL**
   as `papazao://close` (or whatever the carecart admin set under
   `/admin/settings → Partner integration`).
3. Tap **Save** → returns to the home screen.
4. Tap the **Shop** tile. A branded splash holds the screen briefly,
   then carecart loads via `/h5/login?ppzid=…&email=…` — the handoff
   signs the user in automatically.
5. To test the close flow: tap the **Home** icon in the carecart
   mobile bottom nav. The WebView should close and you're back on the
   partner-app home.
6. To test the camera flow: as an admin/manager/vendor, navigate to
   **Manage Collection**. The camera viewport should auto-start with
   no prompts; point it at any customer's collection QR (or paste
   the order number into the search field) and confirm the scan modal
   opens.

## What's actually being tested

### 1. Close-bridge: iOS WKWebView script message handler

Carecart calls:

```js
window.webkit.messageHandlers.closeWebView.postMessage(null)
```

In Flutter we register a JavaScript channel named `closeWebView`:

```dart
..addJavaScriptChannel(
  'closeWebView',
  onMessageReceived: (_) => _close(),
);
```

On iOS, `webview_flutter` publishes the channel at exactly
`window.webkit.messageHandlers.closeWebView`, so carecart's iOS path
hits this handler with no extra glue.

### 2. Close-bridge: Android `window.Android.closeWebView()`

Carecart's Android path (the `@JavascriptInterface` convention) calls:

```js
window.Android.closeWebView()
```

Flutter's channel API only exposes `window.<name>.postMessage(...)`,
not arbitrary methods on a named object, so we inject a tiny shim on
every `onPageFinished` that defines `window.Android.closeWebView` to
forward to the same Flutter channel.

### 3. Close-bridge: URL-scheme deep link

If neither bridge is registered, carecart falls back to
`window.location.href = '<closeUrl>'` (defaults to `papazao://close`).
We catch it in `onNavigationRequest` before the WebView tries to load
the unknown scheme:

```dart
onNavigationRequest: (req) {
  if (close.isNotEmpty && req.url.startsWith(close)) {
    _close();
    return NavigationDecision.prevent;
  }
  return NavigationDecision.navigate;
},
```

### 4. Camera access for the Manage Collection scanner

When admin/manager/vendor open `/admin/collection` (or
`/vendor/collection`) and the page calls
`navigator.mediaDevices.getUserMedia`, two things have to be true on
the host side:

**Runtime permission for the host process.** Manifest declaration
alone isn't enough on Android 6+ — the OS still requires a runtime
grant. We call `Permission.camera.request()` in `initState`:

```dart
if (Platform.isAndroid) {
  Permission.camera.request();
}
```

**Per-page WebView permission grant.** Android's WebView denies every
page-level camera request by default. We hook
`AndroidWebViewController.setOnPlatformPermissionRequest` and grant
camera/microphone:

```dart
if (controller.platform is AndroidWebViewController) {
  final android = controller.platform as AndroidWebViewController;
  android.setMediaPlaybackRequiresUserGesture(false);
  android.setOnPlatformPermissionRequest((request) {
    request.grant();
  });
}
```

iOS WKWebView shows a native camera prompt automatically when the
page calls `getUserMedia`, provided `NSCameraUsageDescription` is
present in `Info.plist` (it is — see
[`ios/Runner/Info.plist`](ios/Runner/Info.plist)).

The production partner app needs the equivalent: CAMERA in the
manifest, the runtime permission grant, and either
`onPermissionRequest` (native Android `WebChromeClient`) or
`setOnPlatformPermissionRequest` (Flutter) to grant the page's
request.

### Camera permission

Already wired in this repo:

- **Android manifest** — [`android/app/src/main/AndroidManifest.xml`](android/app/src/main/AndroidManifest.xml)
  has `<uses-permission android:name="android.permission.CAMERA"/>`
  and `<uses-feature android:name="android.hardware.camera" android:required="false"/>`.
- **iOS plist** — [`ios/Runner/Info.plist`](ios/Runner/Info.plist) has
  `NSCameraUsageDescription`.

If you regenerate `android/` or `ios/` from scratch with
`flutter create`, re-add those before testing the scanner.

## Splash overlay

`webview_screen.dart` paints a branded white overlay (carecart icon +
spinner) on top of the WebView while it's loading and fades it out
once the page paints. Without this the WebView renders white during
navigation, which reads as the app being frozen. Mostly cosmetic,
but worth keeping in production for the same reason.

## Troubleshooting

- **"Tap to open Shop" does nothing** → you haven't saved a PPZ ID +
  email in Settings. Snackbar should tell you.
- **WebView shows a partner-API error page** → either the PPZ ID
  isn't in the partner system, the email doesn't match the partner
  record, or carecart's `PPZ_API_KEY` env var isn't set. Open the
  URL on a desktop browser to see the carecart `/login?error=<code>`
  redirect.
- **Home button on carecart doesn't close the WebView** → the
  configured Close URL doesn't match. Carecart admin sets it under
  `/admin/settings → Partner integration`. Whatever's there must
  equal what you typed in the sample app's Settings.
- **Manage Collection: "Camera permission was blocked" or "Could
  not start video source"** → the runtime camera grant is the most
  common gotcha. Walk these in order:
  1. **Manifest entry** — `android/app/src/main/AndroidManifest.xml`
     contains `<uses-permission android:name="android.permission.CAMERA"/>`.
     (It does in this repo. If you regenerated `android/` from scratch,
     re-add it.)
  2. **Runtime permission grant** — Settings → Apps →
     `carecart_partner_sample` → Permissions → Camera → set to
     **Allow**. The app calls `Permission.camera.request()` in
     `initState`, so a fresh install prompts automatically; if you
     previously denied the prompt it won't reappear and you have to
     flip it manually.
  3. **Re-install after manifest edits** — hot-restart doesn't pick
     up new manifest entries. Stop the app fully and `flutter run`
     again.
  4. **Cross-check in standalone Chrome** — open the same URL in
     real Chrome on the same phone. If the camera works there, the
     issue is on the host-app side; if it fails there too, the
     device camera itself is the problem.
- **Can't see network traffic** → use Chrome DevTools at
  `chrome://inspect/#devices` while the emulator is running and the
  WebView is loaded. You'll see all the requests and console logs.

## Once you're satisfied

Hand the production iOS / Android team:

- The carecart **base URL**.
- The **H5 handoff URL pattern**: `/h5/login?ppzid=…&email=…`.
- The **Close URL value** from `/admin/settings → Partner integration`.
- Confirmation of which **close path** their stack will use (iOS
  message handler `closeWebView`, Android JS interface
  `Android.closeWebView`, or URL scheme) — this sample proves all
  three work, so the choice is theirs.
- The **camera-access checklist** for `/admin/collection`:
  - Android: `CAMERA` in manifest, runtime permission grant on
    first launch, and `WebChromeClient.onPermissionRequest` (native)
    or `setOnPlatformPermissionRequest` (Flutter) granting camera/mic.
  - iOS: `NSCameraUsageDescription` in `Info.plist` — WKWebView
    prompts automatically.
- The relevant interception snippets from
  [`webview_screen.dart`](lib/webview_screen.dart) (Flutter) or the
  carecart README's *Partner integration* section (Swift / Kotlin
  equivalents).
